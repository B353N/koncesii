# Архитектура

Преглед на системата на едно място и указател към записаните решения. **Архитектурните решения
живеят като ADR в [`adr/`](adr/README.md)** — този файл е картата, не източникът на истина за
отделните решения. Продъкшънът работи изцяло на собствения сървър, управляван с Coolify
([ADR-0005](adr/0005-selfhosted-coolify.md)); базовите конвенции на монорепото са от
[ADR-0001](adr/0001-cloudflare-monorepo-stack.md) (заменен за хостинг частта).

## Поток на данните

```
НКР (nkr.government.bg) ──────┐
Стар НКР (nkrold) ────────────┤   tools/harvest (Python, BG IP, bootstrap)
data.egov.bg API ─────────────┼──▶ + apps/etl (планиран job, инкрементален refresh)
МЕ / МТ регистри ─────────────┘        │  raw snapshots (сървъра) → normalize → unify
                                       │  → derive flags → precompute
                                       ▼
                                 koncesii.sqlite (един файл на среда)
                                       ▲
                                       │  read-only
              граждани / журналисти ──▶  koncesii (RR SSR, Node контейнер в Coolify)
```

Данните пристигат като **периодични извличания**, не като жива емисия. Регистрите нямат
webhook-ове и режат datacenter IP-та, затова извличането е двустепенно
([ADR-0002](adr/0002-two-stage-ingest.md)):

1. **Bootstrap** — Python инструментите в `tools/harvest/` (пълно обхождане на НКР, стария
   регистър и data.egov.bg) се пускат ръчно от българско IP и качват суровия резултат в
   snapshot хранилището на сървъра (`/data/koncesii/snapshots/YYYY-MM-DD/…`).
2. **Инкрементален refresh** — планираният job (`apps/etl`) тегли новите/променените партиди
   (експортът на НКР е една заявка от ~500 KB) през BG residential proxy или self-hosted
   runner и произвежда нов SQLite файл.

ETL-ът произвежда нов `koncesii.sqlite`; explorer-ът чете текущия, подмяната е **атомарна**
(запис до временно име + `mv`). В v1 няма публичен write път и няма автентикация.

## Сървисите (Coolify)

| Сървис     | Роля                                                                          |
| ---------- | ----------------------------------------------------------------------------- |
| `koncesii` | SSR explorer (`apps/web`), Node контейнер — чете `koncesii.sqlite` read-only. |
| `apps/etl` | планиран refresh job (cron/scheduled task, фаза 5) — произвежда новия SQLite. |

Деплой: мърдж към `main` → GitHub webhook → Coolify билдва (nixpacks) и подменя контейнера.

## Слоеве на данните

| Слой    | Къде живее                           | Съдържание                                                        |
| ------- | ------------------------------------ | ----------------------------------------------------------------- |
| raw     | snapshot хранилището на сървъра      | сурови HTML/TSV/JSON от източниците, immutable, с дата на снемане |
| staging | `raw_*` таблици в koncesii.sqlite    | парснати записи 1:1 с източника, с `source` префикс               |
| domain  | domain таблици в koncesii.sqlite     | единният модел: concessions / grantors / concessionaires / …      |
| derived | `flags`, `rollups` в koncesii.sqlite | изчислените индикатори и precompute агрегати                      |

Единният модел и dedup правилата между източниците (една концесия често съществува и в НКР,
и в общинския регистър на data.egov.bg) — [`core-scope.md`](core-scope.md) и
[ADR-0003](adr/0003-nkr-wins-dedup.md).

## Рендиране и сигурност (резюме)

Пре-рендиране за статичните страници, SSR + HTTP кеш за
детайлните/explorer страници, клиентски „острови" за интерактивното (картата е най-големият
остров — MapLibre GL, [ADR-0004](adr/0004-map-rendering.md)). Приоритет интегритет ≈
наличност ≫ конфиденциалност (данните са публични). Read пътят е read-only; строг CSP;
SQLite само с bound параметри.

## Индикаторите за риск

Флаговете са детерминистични, изчисляват се в ETL derive стъпката и имат публична методология
в [`red-flags.md`](red-flags.md). Никога не се изчисляват в заявка на живо — потребителят
вижда точно това, което методологията описва, върху точно тази версия на данните.
