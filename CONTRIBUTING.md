# Принос към КОНЦЕСИИ

Благодарим, че искате да допринесете! 🙏 КОНЦЕСИИ е портал за прозрачност на концесиите
и всеки принос — поправка, тест, данни или идея — е добре дошъл.

Този документ е кратък наръчник за работния поток. Канонична за конвенциите (комити,
клонове, граници на промяната) е [`AGENTS.md`](AGENTS.md); архитектурата и решенията живеят
в [`docs/`](docs/), а бързият старт — в [`README.md`](README.md).

## Преди да започнете

- За **бъг, проблем с данните или идея** — отворете issue по подходящия шаблон. Преди това
  проверете дали вече няма такъв. Проблемите с данните (грешна сума, липсваща концесия,
  сгрешен концесионер) са отделен шаблон — те са най-ценният принос от нетехнически хора.
- За **уязвимост в сигурността** — **не** отваряйте публичен issue. Вижте раздела **Security**
  на хранилището.
- За по-голяма промяна е добре първо да я обсъдим в issue, преди да отделяте време за код.

## Среда за разработка

Хранилището е пригодено за **Devcontainer** (нужни са Docker или съвместим и редактор с
поддръжка). Изисквания извън контейнера: Node `>=22` и `pnpm` (виж `packageManager` в
`package.json`).

```bash
pnpm setup    # еднократно: инсталиране + локален D1 + примерни данни
pnpm dev      # ежедневно: приложението + ETL worker-а паралелно
```

Bootstrap извличането от регистрите (`tools/harvest/`) изисква **българско IP** — държавните
сайтове режат datacenter адреси. За разработка не ви трябва: `pnpm setup` зарежда примерни
данни от фикстури.

## Работен поток (форк + PR)

КОНЦЕСИИ е trunk-based: `main` е единственият дълготраен клон, приносите идват през **форк и PR**.

1. **Форкнете** хранилището и клонирайте вашия форк.
2. Създайте клон от най-новия `main` с име `<type>/<slug>` — напр. `fix/payment-parse-eur`,
   `docs/red-flags`. `<type>` е като при комитите по-долу.
3. Направете промяната — малки, фокусирани комити, докато напредвате (не всичко накрая).
4. Пуснете проверките (виж по-долу) и **отворете PR към `main`** от вашия форк.
5. Един PR = една логическа промяна, за да остане лесен за преглед.

## Комити

Следваме [conventional commits](https://www.conventionalcommits.org):
`<type>(<scope>): <subject>`.

- Типове: `feat`, `fix`, `docs`, `refactor`, `test`, `build`, `ci`, `chore`, `perf`, `style`.
- Scope-ове по workspace: `web`, `etl`, `db`, `ingest`, `shared`, `harvest`, `docs`.
- Темата е в малки букви, повелително наклонение, без точка накрая. Пример:
  `fix(ingest): decode nkr export as windows-1251`.
- **Без `Co-Authored-By:` trailer-и** — историята се пази чиста (CI може да ги проверява).

## Проверки преди PR

Пуснете само нужния минимум, за да сте уверени в промяната (пълна проверка се прави при
release):

```bash
pnpm typecheck      # типове
pnpm test           # тестове (или само засегнатия пакет: pnpm --filter <pkg> test)
pnpm lint           # форматиране (Prettier); pnpm format поправя
```

## Какво да не правите

- Не комитвайте тайни, `.env*` или `.dev.vars`. Не комитвайте и **сурови данни** от
  извличането (HTML дъмпове, TSV експорти) — те живеят в R2, не в git.
- Не amend-вайте и не force-push-вайте вече публикуван клон, който някой може да чете.
- Не трийте клонове, които не сте създали.
- Не пипайте файлове извън обхвата на промяната. Намерите ли несвързан проблем — отбележете
  го в отделен issue.
- Не претоварвайте държавните регистри: 1 заявка/секунда, честен User-Agent, resumable jobs.
- Дизайнерски решения и планове — в [`docs/`](docs/), не като разпръснати бележки в кода.
- Никога не формулирайте флаг като обвинение. Флагът е възпроизводим аритметичен факт с
  публична методология ([`docs/red-flags.md`](docs/red-flags.md)); текст, който внушава
  нарушение, е дефект.

## Стил и език

- Пишете код в стила на околния (именуване, коментари, идиоми).
- Текстовете към потребителя са на български. Дизайнерската проза в `docs/` е на български;
  кодът, схемите и комитите — на английски.
